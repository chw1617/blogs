# JS 作用域链

# 作用域链

  • 函数的作用域：当查找变量的时候，会先从当前的上下文执行栈的变量对象中查找，如果没有找到，就从父级（词法上的作用域）的上下文的变量对象中查找，直到查找到全局上下文的变量对象，这样由多一个执行上下文变量对象组成的链表就是作用域链
  • 下面我们从函数定义和执行的时候，作用域链的变化是怎么样的

# 函数定义时

我们在js词法作用域分析的时候就已经解析过，函数在定义的时候就已经确定了函数的作用域

  • 这个是因为函数有一个[[scope]]属性，函数定义的时候，保存了所有的父变量对象
  • 下面demo可以看看
function f1(){
    function f2(){}
}
f1()
解析：
函数f1定义时：f1.[[scope]] = [globalContext.vo]
函数f1执行时候：f2定义： f2.[[scope]] = [f1Context.AO,globalContext.vo]

# 函数执行时

还是上面的例子

function f1(){
    function f2(){}
}
f1()
解析：
函数f1执行时候：f2定义： f2.[[scope]] = [f1Context.AO,globalContext.VO]
f1的scope变成：f1.[[scope]] = [f1Context.AO,globalContext.VO]

# 具体综合分析

还是直接具体分析代码，

  • 解析js的执行过程中作用域链、执行上下文栈、变量对象的变化
  • 闭包一直都是作用域的问题，这个也会讲解到闭包的产生（函数定义的时候就将上层的作用域保存到[[scope]],同时上层作用域的变量）
var scope = 'global scope'
function f1(){
    var scope = 'local scope'
    function f2(){
        return scope
    }
    return f2
}
f1()()

1、这段代码执行，先创建全局的上下文栈，这个时候上下文栈为

ECStack = [globalContext]

2、进入全局的上下文的同时，函数f1也被定义，f1的scope会将上层的作用域链到内部[[scope]]

globalContext = {
    VO:{
        scope,
        f1
    },
    scope:[globalContext.VO],
    this:globalContext.VO
}

f1.[[scope]] = [globalContext.VO]

3、执行函数f1，创建f1的执行上下文，并入栈

ECStack = [f1Context,globalContext]

4、进入函数f1的执行上下文，创建f1.AO,同时函数f2也被定义,并将父作用域链保存到[[scope]]

f1Context = {
    AO:{
        arguments:{},
        scope:undefiend,
        f2:function(){}
    },
    scope:[AO,globalContext.VO],
    this:undefined
}

f2.[[scope]] = [f1Context.AO,globalContext.VO]（闭包根本）

5、f1函数开始执行，改变f1上下文的AO,同时将f2返回到外部(从而产生闭包)

f1Context = {
    AO:{
        arguments:{},
        scope:'local scope',
        f2:function(){}
    },
    scope:[AO,globalContext.VO],
    this:undefined
}

6、f1执行完成，退出上下文执行栈

ECStack = [globalContext]

7、函数f2接着执行，开始进入f2的执行栈

ECStack = [f2Context,globalContext]
f2Context = {
    AO:{
        arguments:{},
    },
    scope:[AO,f1Context.AO,globalContext.VO],
    this:undefined
}

8、f2执行,自己AO中没有找到，沿这作用域查找scope为local scope，最后退出上下文栈

ECStack = [globalContext]

# 总结

1、函数定义时候：保存了父级（书写的位置）的作用域链
2、函数执行时候：改变活动对象、作用链、上下文栈
